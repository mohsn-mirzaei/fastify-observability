بیا گام‌به‌گام پرومتئوس رو یاد بگیریم، از صفر تا جایی که بتونی خودت راهش بندازی و باهاش کار کنی. سعی می‌کنم واقعاً «آدمیزادی» توضیح بدم، انگار کنار هم نشستیم و داریم حرف می‌زنیم.

---

## ۱. اصلاً پرومتئوس چیه و چرا به درد می‌خوره؟

فرض کن یه سرویس داری (وب‌سایت، API، دیتابیس، سرور و …). می‌خوای بدونی:

- پردازنده سرور چقدر مشغوله (CPU usage)
- رم چقدر مصرف شده
- تعداد درخواست‌های وب‌سایتت در ثانیه چقدره
- چه تعدادشون خطا خوردن (ارور ۵۰۰)
- دیسک چقدر فضای خالی داره
  و هزارتا سوال دیگه از این جنس.

**پرومتئوس** یه سیستم جمع‌آوری و ذخیره‌سازیِ داده‌های عددیِ سری‌زمانی (Time Series) و هشدار دهی (Alerting) هست. به زبون ساده، یه ابزار نظارت (Monitoring) که مدام می‌ره از سرویس‌هات آمار می‌گیره (مثلاً هر ۱۵ ثانیه)، این آمار رو تو یه دیتابیس مخصوص خودش می‌ریزه، بعداً می‌تونی با یه زبان پرس‌وجو (PromQL) ازش سوال بپرسی و نمودار بکشی یا اینکه شرط بذاری «اگه CPU از ۹۰٪ رفت بالاتر، بهم خبر بده».

**ویژگی‌های کلیدی که خیلی مهمه:**

- **Pull (کشیدن):** پرومتئوس خودش میره سراغ هدف‌ها (Targets) و اطلاعات رو می‌کشه. برعکس خیلی از ابزارها که هدف‌ها باید اطلاعات رو push کنن. این کار سادگی و کنترل رو بالا می‌بره.
- **مدل داده بر پایه لیبل (Label):** داده‌هات فقط یه عدد نیستن؛ کلی برچسب روشون می‌زنی. مثلاً متریک `http_requests_total` می‌تونه لیبل‌های `method="GET"`, `status="200"`, `endpoint="/api"` داشته باشه. بعداً می‌تونی راحت بر اساس این لیبل‌ها فیلتر و جمع بزنی.
- **PromQL:** یه زبان پرس‌وجوی ساده و فوق‌العاده قدرتمند برای کار با متریک‌ها.
- **عدم وابستگی به ذخیره‌سازی توزیع‌شده:** هر سرور پرومتئوس مستقل کار می‌کنه، برای آدمای عادی عالیه، دردسر کلاسترهای پیچیده رو نداره. (برای مقیاس‌های خیلی بزرگ هم بعداً راهکار دارن).

---

## ۲. مفاهیم اصلی با یه مثال ملموس

بیا یه مثال بزنیم: می‌خوایم تعداد درخواست‌های HTTP وب‌سرورمون رو ببینیم.

### ۲.۱ متریک (Metric)

متریک یعنی همون چیزی که اندازه‌گیری می‌کنیم، مثلاً:

- `http_requests_total` (تعداد کل درخواست‌ها)

### ۲.۲ لیبل‌ها (Labels)

روی هر متریک می‌تونیم یه سری برچسب بچسبونیم تا بعداً جزئیات رو فیلتر کنیم. مثلاً متریک بالا می‌تونه این لیبل‌ها رو داشته باشه:

- `method`: GET, POST, ...
- `status`: 200, 404, 500, ...
- `endpoint`: /api/users, /api/login

پس یه نمونه داده این شکلیه:

```
http_requests_total{method="GET", status="200", endpoint="/api/users"}  1523
```

یعنی تا حالا ۱۵۲۳ تا درخواست GET موفق (کد ۲۰۰) به مسیر `/api/users` داشتیم.

### ۲.۳ مدل داده (Data Model)

داده‌ها به صورت **سری زمانی (Time Series)** ذخیره می‌شن:
یه سری زمانی = اسم متریک + یه مجموعه مشخص از لیبل‌ها + یه دنباله از (مقدار، زمان).

مثلاً سری زمانی بالا هر چند ثانیه یه دونه عدد داره (که اینجا چون از نوع شمارنده هست، مدام زیاد می‌شه).

### ۲.۴ انواع متریک (Metric Types)

پرومتئوس ۴ نوع متریک اصلی داره که خیلی مهمه درکشون کنی:

- **Counter (شمارنده):** فقط زیاد می‌شه (یا ریست می‌شه به صفر، مثلاً با ری‌استارت شدن سرویس). مثل تعداد کل درخواست‌ها، تعداد ارورها، تعداد بایت‌های ارسالی. برای نرخ (rate) استفاده می‌شه، نه مقدار مطلق.
- **Gauge (گیج):** کم و زیاد می‌شه، مثل دمای CPU، مصرف رم، تعداد کاربران آنلاین. برای مقدار لحظه‌ای.
- **Histogram (هیستوگرام):** مشاهدات رو توی دسته‌های از پیش تعریف شده می‌ریزه. مثلاً زمان پاسخ‌دهی درخواست‌ها: چندتا زیر ۰.۱ ثانیه، چندتا بین ۰.۱ تا ۰.۵ ثانیه و … . به‌کمکش می‌تونی صدک (percentile) حساب کنی.
- **Summary (خلاصه):** شبیه هیستوگرام ولی صدک‌ها رو سمت کلاینت (اپلیکیشن) حساب می‌شه، نه سمت پرومتئوس. هیستوگرام منعطف‌تره و معمولاً توصیه می‌شه.

## ۳. معماری کلی (نقشه راه)

آدمیزادی‌ترین دید اینه:

1. **سرور پرومتئوس (Prometheus Server):** مغز ماجرا، مسئول کشیدن داده‌ها، ذخیره‌سازی و ارزیابی قوانین.
2. **هدف‌ها (Targets):** سیستم‌هایی که ازشون داده می‌کشیم، مثلاً خود اپلیکیشن‌ها، سرورهای لینوکسی، دیتابیس‌ها. اینا باید یه endpoint HTTP بدهن که متریک‌ها رو با یه فرمت خاص برگردونه (معمولاً path مثل `/metrics`).
3. **Exporters (صادرکننده‌ها):** وقتی یه سرویس خودش متریک‌های Prometheus رو بلد نیست، یه ابزار کمکی کنارش می‌گذاریم که اطلاعات رو ترجمه کنه. مثلاً `node_exporter` برای متریک‌های سیستم‌عامل (CPU, RAM, Disk).
4. **Alertmanager:** مسئول مدیریت هشدارها. پرومتئوس هشدار رو می‌فرسته به Alertmanager و اون کارای group کردن، silencing، ارسال به ایمیل/اسلک/پیامک و ... رو انجام می‌ده.
5. **Grafana (یا ابزارهای بصری):** به پرومتئوس وصل می‌شه و داشبوردهای قشنگ می‌سازه. خود پرومتئوس یه UI ساده داره، ولی گرافانا حرفه‌ای‌تره.

نمودار جریان:

```
[Exporters/Targets]  <--- (pull metrics over HTTP) --- [Prometheus Server]
                                                      |
                                                      v
                                           [Alertmanager] -> Email/Slack/...
```

و بعداً Grafana از Prometheus کوئری می‌گیره.

## ۴. نصب و راه‌اندازی (فقط چند دقیقه وقت می‌خواد)

خیلی ساده‌ست. پرومتئوس با Go نوشته شده و یه فایل اجرایی binary هست، نیاز به نصب پیچیده نداره.

۱. برو به [prometheus.io/download](https://prometheus.io/download/) و نسخه‌ی مناسب سیستم‌عاملت رو بگیر (لینوکس، مک، ویندوز).  
۲. فایل فشرده رو باز کن، یه پوشه می‌سازی با محتویات:

- `prometheus` (فایل اجرایی)
- `prometheus.yml` (فایل پیکربندی پیش‌فرض)
- `console_libraries`, `consoles` (برای UI قدیمی، نادیده بگیر)
  ۳. برای تست، مستقیم `./prometheus` رو اجرا کن. (تو ویندوز `prometheus.exe`).  
  ۴. حالا تو مرورگر برو به `http://localhost:9090`. باید UI پرومتئوس رو ببینی.  
  ۵. تو تب Status > Targets می‌تونی ببینی پرومتئوس خودش رو به عنوان هدف scrape می‌کنه (آدرس `localhost:9090/metrics`). یعنی متریک‌های خودش رو هم می‌تونی ببینی.

**نکته:** برای محیط واقعی، پرومتئوس رو به عنوان سرویس systemd یا داکر اجرا کن. ولی فعلاً همین کافیه.

## ۵. تنظیمات فایل `prometheus.yml` (قلب پیکربندی)

بازش کن، محتوای پیش‌فرضش این شکلیه:

```yaml
global:
  scrape_interval: 15s # هر چند وقت یه بار بره سراغ هدف‌ها (پیش‌فرض ۱۵ثانیه)
  evaluation_interval: 15s # هر چند وقت قوانین هشدار رو چک کنه

scrape_configs:
  - job_name: "prometheus"
    static_configs:
      - targets: ["localhost:9090"]
```

- `global`: تنظیمات عمومی. scrape_interval یعنی هر چند ثانیه یه بار متریک‌ها رو بکشه. می‌تونی بعداً برای هر job جداگانه تنظیم کنی.
- `scrape_configs`: اینجا می‌گی کجاها رو بگرده.
- `job_name`: یه اسم برای این کار (می‌تونه هرچی باشه، مثل `node`, `myapp`).
- `static_configs`: لیست ثابتی از هدف‌ها. هر کدوم می‌تونه یه `host:port` باشه. پرومتئوس می‌ره به `http://host:port/metrics`.

بیا یه job جدید برای سیستم‌عامل اضافه کنیم. اول باید `node_exporter` رو اجرا کنیم (از همون سایت دانلود کن، اجرا کن، پیش‌فرض روی پورت ۹۱۰۰).

حالا به `prometheus.yml` اضافه کن:

```yaml
- job_name: "node"
  static_configs:
    - targets: ["localhost:9100"]
```

پرومتئوس رو ری‌استارت کن، بعد تو UI ببین targets، الان node رو up می‌بینی. می‌تونی متریک‌هایی مثل `node_cpu_seconds_total` رو کوئری بزنی.

## ۶. جمع‌آوری داده از جاهای مختلف (Exporters و Instrumentation)

### ۶.۱ اگر سرویست متریک نداده:

کافیه یه exporter متناسب با اون تکنولوژی بذاری کنارش. چندتا معروف:

- **Node Exporter:** آمار CPU، مموری، دیسک، شبکه و … سرور لینوکسی/ویندوزی.
- **Blackbox Exporter:** بررسی کنه یه سایت از بیرون زنده‌ست یا نه (probe HTTP, DNS, TCP).
- **MySQL Exporter, Redis Exporter, PostgreSQL Exporter** و …
- **cAdvisor** برای کانتینرهای داکر.

### ۶.۲ اگه خودت اپلیکیشن می‌نویسی:

کتابخونه‌های کلاینت پرومتئوس برای زبان‌های مختلف وجود داره (Go, Java, Python, Node.js, …). با چند خط کد یه endpoint مثلاً `/metrics` درست می‌کنی که متریک‌هایی که خودت تعریف کردی رو برگردونه. بعد تو پرومتئوس scrape‌ش کن.

**مثال ساده در پایتون با کتابخونه `prometheus_client`:**

```python
from prometheus_client import start_http_server, Counter
import time

REQUESTS = Counter('myapp_http_requests_total', 'Total HTTP requests', ['method', 'endpoint'])

@route('/api')
def api():
    REQUESTS.labels(method='GET', endpoint='/api').inc()
    return "OK"

start_http_server(8000)  # متریک‌ها روی پورت 8000 سرو می‌شن
```

حالا تو `prometheus.yml` این هدف رو اضافه کن `localhost:8000`.

## ۷. PromQL؛ پرسیدن سوال از پرومتئوس به زبان آدمیزاد

حالا که داده جمع شده، چطور تحلیلش کنیم؟ تو تب Graph در UI پرومتئوس می‌تونی عبارات PromQL بنویسی.

### مفاهیم پایه:

- **Instant Vector (بردار لحظه‌ای):** یه مجموعه از سری‌های زمانی در یه لحظه خاص. مثلاً `http_requests_total` همه سری‌هایی که این اسم رو دارن برمی‌گردونه با آخرین مقدارشون.
- **Range Vector (بردار بازه‌ای):** مجموعه‌ای از مقادیر در یک بازه زمانی. مثلاً `http_requests_total[5m]` برمی‌گردونه آرایه‌ای از مقادیر ۵ دقیقه اخیر برای هر سری. اینو بیشتر با توابع به کار می‌بریم.

### توابع معروف (با مثال کاربردی):

۱. **`rate()`:** نرخ رشد یه Counter در هر ثانیه رو حساب می‌کنه (مناسب برای محاسبه "درخواست در ثانیه").

```
rate(http_requests_total[5m])
```

یعنی میانگین تعداد درخواست در ثانیه در ۵ دقیقه گذشته.

۲. **`irate()`:** نرخ لحظه‌ای، حساس‌تر به تغییرات اخیر. برای آلرت‌های سریع گاهی بهتره.

```
irate(http_requests_total[1m])
```

۳. **`sum()`:** جمع کردن مقادیر چند سری. معمولاً با `by` یا `without` برای تجمیع روی لیبل‌ها:

```
sum(rate(http_requests_total[5m])) by (status)
```

تعداد درخواست در ثانیه به تفکیک کد وضعیت.

۴. **`avg()`:** میانگین گرفتن.

```
avg(rate(node_cpu_seconds_total{mode="idle"}[5m])) by (instance)
```

۵. **`histogram_quantile()`:** برای محاسبه صدک از هیستوگرام.

```
histogram_quantile(0.95, sum(rate(http_request_duration_seconds_bucket[5m])) by (le))
```

یعنی ۹۵٪ درخواست‌ها در چه زمانی پاسخ گرفتن.

**فیلتر با لیبل‌ها:**

- `{job="node", instance="localhost:9100"}`: انتخاب با لیبل
- عملگرها: `=`, `!=`, `=~` (regex match), `!~`.
  مثلاً `http_requests_total{status=~"5.."}` کدهای ۵۰۰ تا ۵۹۹.

## ۸. قوانین ضبط (Recording Rules)

گاهی یک کوئری سنگین رو مکرر اجرا می‌کنیم. می‌تونیم نتیجه رو از قبل محاسبه و ذخیره کنیم تا سریع‌تر بشه. این کار با **Recording Rules** انجام می‌شه.

یه فایل `rules.yml` بساز با این محتوا:

```yaml
groups:
  - name: example
    rules:
      - record: job:http_requests_total:rate5m
        expr: rate(http_requests_total[5m])
```

تو `prometheus.yml` اضافه کن:

```yaml
rule_files:
  - "rules.yml"
```

حالا متریک جدیدی به اسم `job:http_requests_total:rate5m` خواهی داشت که هر بار ارزیابی می‌شه و می‌تونی مثل یه متریک معمولی ازش کوئری بگیری. عالیه برای داشبوردها.

---

## ۹. هشداردهی (Alerting) – وقتی اوضاع خراب شد خبرم کن

هشدار دو بخش داره: **تعریف شرط در پرومتئوس** و **مدیریت هشدار با Alertmanager**.

### ۹.۱ قوانین هشدار (Alerting Rules)

تو همون فایل `rules.yml` (یا جدا)، می‌نویسی:

```yaml
groups:
  - name: alerts
    rules:
      - alert: HighCPUUsage
        expr: 100 - (avg(rate(node_cpu_seconds_total{mode="idle"}[5m])) by (instance) * 100) > 80
        for: 5m # حداقل ۵ دقیقه این وضعیت طول بکشه
        labels:
          severity: critical
        annotations:
          summary: "CPU usage > 80% on {{ $labels.instance }}"
          description: "CPU usage is {{ $value }}% for 5 minutes."
```

- `expr`: شرط، اینجا اگه درصد استفاده CPU از ۸۰ بیشتر بشه.
- `for`: مدت زمانی که شرط باید پایدار بمونه تا هشدار فعال بشه (از flickering جلوگیری می‌کنه).
- `labels`: لیبل‌هایی که به هشدار اضافه می‌شه (برای مسیریابی در Alertmanager).
- `annotations`: توضیحات اضافه، معمولاً قالب‌بندی با متغیرها.

وقتی فعال بشه، تو UI پرومتئوس در بخش Alerts می‌بینی.

### ۹.۲ Alertmanager

پرومتئوس به تنهایی فقط هشدارها رو تولید و ارزیابی می‌کنه، اما ارسالشون رو می‌سپره به Alertmanager. Alertmanager کاراش:

- **Grouping:** هشدارهای مشابه رو دسته‌بندی کنه (مثلاً چند تا سرور همشون CPU بالا دارن، یه ایمیل بفرسته نه ۱۰ تا).
- **Inhibition:** وقتی یه هشدار سطح بالا اومد، هشدارهای مرتبط سطح پایین رو متوقف کنه (مثلاً وقتی کل دیتاسنتر داون شده، هشدار تک‌تک سرورها فایده نداره).
- **Silencing:** موقتاً هشدارها رو برای تعمیرات خاموش کنه.
- **Routing:** بر اساس لیبل‌ها به مقصدهای مختلف (ایمیل، اسلک، PagerDuty، Webhook و …) بفرسته.

نصب Alertmanager مثل پرومتئوس ساده‌ست (دانلود، کانفیگ، اجرا). یه فایل `alertmanager.yml` داری:

```yaml
global:
  resolve_timeout: 5m

route:
  receiver: "team-slack"
  group_by: ["alertname"]
  group_wait: 30s
  group_interval: 5m
  repeat_interval: 4h

receivers:
  - name: "team-slack"
    slack_configs:
      - api_url: "https://hooks.slack.com/services/..."
        channel: "#alerts"
```

تو پرومتئوس هم آدرس Alertmanager رو می‌دی:

```yaml
alerting:
  alertmanagers:
    - static_configs:
        - targets: ["localhost:9093"]
```

وقتی پرومتئوس هشداری تشخیص بده، به Alertmanager پوش می‌کنه، اون هم طبق routing می‌فرسته.

## ۱۰. زیباسازی با Grafana

پرومتئوس یه تب Graph داره، ولی Grafana رو همه دوست دارن. نصبش ساده‌ست (پکیج یا داکر). بعد:

1. Data source از نوع Prometheus اضافه کن، آدرس `http://localhost:9090`.
2. یه داشبورد بساز، یه پنل بردار، کوئری PromQL بزن.
3. می‌تونی از داشبوردهای آماده (از grafana.com/dashboards) import کنی. مثلاً Node Exporter Full یه داشبورد بی‌نظیر داره.

نمودارهای قشنگ، متغیر برای فیلتر کردن، auto refresh و ... رو بهت میده.

## ۱۱. نکات و روش‌های خوب (Best Practices) برای مبتدی‌ها

- **نام‌گذاری متریک‌ها:** از قاعده `<namespace>_<metric>_<unit>` پیروی کن، مثلاً `myapp_http_requests_total`. اسم Counter به `_total` ختم بشه.
- **لیبل‌های با کاردینالیتی بالا استفاده نکن:** لیبل `user_id` رو روی متریک قرار نده، چون ممکنه میلیون‌ها سری زمانی تولید کنه و پرومتئوس رو منفجر کنه. همیشه اطلاعاتی رو که می‌خوای aggregate کنی بذار توی لیبل‌ها (مثلاً `status`, `method`).
- **scrape interval مناسب:** ۱۵-۳۰ ثانیه برای اکثر کارها خوبه. برای بعضی متریک‌های حساس هم ۵ ثانیه.
- **متریک‌های مهم رو پایش کن:** چهار سیگنال طلایی (USE + RED):
  - USE برای منابع (Utilization, Saturation, Errors): CPU, Mem, Disk, Network.
  - RED برای سرویس‌ها (Rate, Errors, Duration): نرخ درخواست، خطاها، مدت‌زمان پاسخ.
- **همیشه `rate` رو روی Counter با بازه معقول بگیر:** مثلاً `rate(x_total[5m])` نه بازه خیلی کوتاه که نویز بده، نه خیلی بلند که تغییرات رو دیر نشون بده.
- **برای HA چطور؟** پرومتئوس خودش clustered نیست. برای دسترسی بالا و ذخیره طولانی‌مدت می‌تونی بعداً از Thanos یا Cortex استفاده کنی، ولی نگران نباش، توی فاز شروع هیچ لزومی نداره.

## ۱۲. یه خلاصه‌ی گام‌ها برای راه‌اندازی یه مانیتورینگ ساده

1. پرومتئوس رو دانلود و از `prometheus.yml` یه scrape برای node_exporter و اپ خودت اضافه کن.
2. node_exporter رو روی سرور اجرا کن.
3. پرومتئوس رو اجرا کن، ببین targets سبز شدن.
4. چندتا کوئری PromQL تو کنسول بزن تا دستت راه بیفته.
5. قوانین هشدار تعریف کن، مثلاً CPU بالا.
6. Alertmanager رو راه بنداز و وصلش کن به اسلک یا ایمیل.
7. Grafana رو نصب کن، Data Source رو به پرومتئوس بده و یک داشبورد زیبا وارد کن.
8. لذت ببر که همه چیز زیر نظره :))
